What is this? A blob tileset uses 47 unique tiles to auto-tile terrain. Each tile handles a specific combination of filled/empty neighbors so edges, corners, and transitions all connect seamlessly.

Reading the template: The checkerboard background marks tile boundaries — adjacent tiles alternate between two shades of gray so you can see where each tile slot begins and ends. Inside each slot, a 3×3 bitmask indicator shows which neighbors are present: the center cell (always red) is the tile’s own position, the 8 surrounding cells represent the 8 adjacent directions. Red = a tile of the same terrain type (e.g. another wall tile, another grass tile) exists in that direction, white = no matching tile there (edge, corner, or open space). So a tile with only the top three cells red is a bottom edge piece — it has matching terrain above but nothing below.

Variations: The global default sets how many slots each tile gets (1 = base tile only, no extras). Override specific categories to give them more or fewer slots. For example, center tiles and straight edges often benefit from extra variations to avoid visual repetition in large filled areas. Leave an override blank to use the global default.

Layout: Multi-variation groups are packed into rows targeting a square-ish aspect ratio, with colored checkerboard patterns distinguishing each group. Single-variation tiles fill remaining gaps at row ends. Categories appear in order: other (isolated, dead-ends, corridors) → outer corners → straight edges → inner corners → center (fully surrounded).

Workflow: Download the PNG template and duplicate it. Paint your tile art into each slot of the copy — each slot fills one full checkerboard cell edge-to-edge. Keep the original as a reference for setting bitmask values in your game engine.
